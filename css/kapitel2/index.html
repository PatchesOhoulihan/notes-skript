Kapitel 2 CSS verwenden

Einbindungsmöglichkeiten von CSS

Prinizpell drei Möglichkeiten, CSS Gestaltungsvorlagen für ein HTML-Dokument zu definieren:

  1. CSS-Vorgaben werden in einer Separaten Datei erstellt. Die Vorgaben gelten dann für alle verknüpften Seiten, und Änderungen in der CSS Datei wirken sich auf alle Seiten die die Datei linken aus.

  2. CSS-Vorgaben werden im head einer HTML Datei erstellt. Diese gelten dann als CSS-Vorgabe für diese eine Seite.

  3. CSS-Vorgaben werden direkt an einem Tag im Body notiert. Dieses gilt dann nur für das entsprechende Tag.

Diese drei Möglichkeiten können Sie nach Belieben kombinieren.

Die 1. Variante kommt dem Grundgedanken von CSS am nächsten und sollte deshalb im Normalfall verwendet werden.

Allerdings haben die anderen Varianten auch einen Vorteil. Variante 1 und 2 überschreiben die Vorgaben der externen Datei. Dadurch können Sie ihre allgemeinen Vorgaben für eine bestimmte HTML-Seite oder einen Tag überschreiben/ausser Kraft setzen.

Einbinden von CSS-Vorgaben in den Dateikopf

Mit dem <style> Tag definieren Sie in normaler HTML-Syntax den Bereich, in dem Sie Ihre CSS-Vorgaben notieren. Das Attribut type gibt an, welche Stylesheet-Sprache Sie benutzen, als Wert weisen Sie den MIME-Typ von CSS zu: type=text/css

Media Attribut

Das media Attribute ist im Gegensatz zu type KEIN Pflichtattribute

Mit ihm können Sie das Ausgabemedium bestimmen, für das die CSS-Vorgaben gelten sollen.

Möglichkeiten für das media Attribute sind:

  1. media="screen" für die Bildschirm Darstellung.

  2. media="print" vorgaben die für den Druck der Website gelten.

  3. media="aural"(css2). media="speech"(css3) für die Sprachausgabe.

  4. media="all" für alle Medien.

In diesem Fall würden Sie unter style all diejenigen Vorgaben notieren, die für sämtliche Ausgabe-medien gleichermaßen gelten. In den anderen style-Bereichen notieren sie dann die speziellen Vorgaben nur für dieses Ausgabegerät.

Alles, was zwischen <style> und <style> steht, ist CSS-Syntax und folgt eigenen Regeln.

Schriftarten definieren

Klassische Methode

Schriftarten, die Sie auf Ihren Seiten verwenden, müssen auf dem System des Anwenders installiert sein.

Deshalb sollte darauf geachtet werden nur Schriftarten zuverwenden die auf allen Gängigen Betriebsystem vorhanden sind.

  1. Serifenlose Schriften, Verdana, Arial und Helvetica

  2. Serifenschrit, Times und Times New Roman

  3. Festbreitenschrift, Courier

Im Zweifelsfall Verdana verwenden, weil sie die einzige Schrift ist die für die Bildschrimdarstellung optimiert ist.

Sie sollten aber in dem Fall immer mehrere Schriftalternativen in der von Ihnen bevorzugten Reihenfolge anbieten. Der Browser wählt dann immer die erste die er findet.

Schriftnamen, die leerzeichen enthalten, müssen Sie dabei in Anführungszeichen setzen.

Webfonts

Eine andere möglichkeit ist es, die gewünschte Schriftart auf dem Websever zu hinterlegen, sie kann dann bei bedarf heruntergeladen werden.Sie werden erst von neueren Browsern unterstützt.

Browserspezifische Eigenschaften mit Herstellerpräfix.

Irgendwann haben die Brwoserhersteller damit behonnen, neue CSS-Eigenschaften zu unterstützen, die sich noch im experimentaellen Stadium befinden, also noch bevor sie vom W3C offiziell verabschiedet worden sind.

Um jedoch bereits im Vorfeld Konflikte zu vermeiden, die sich bei einer etwailigen Änderung der Spezifikation seitens des W3C ergeben könnten, werden diese experimentellen Features als proprietäre, Browserspezifische Features implementiert, die mit dem sogenannten Vendor Specific Prefix versehen werden müssen.

Wenn sie solche Eigenschaften verwenden, müssen Sie sie bis auf Weiteres mindestens dreimal notieren:

  1. -webkit für Chrome und Safari

  2. -moz für Firefox

  3. -ms für Internet Explorer

Bei einigen Eigenschaften ist auch tatsächlich der Fall eingetreten, dass das W3C zwischenzeitlich die Syntax geändert hat. In diesem Fall müssen Sie die Eigenschaften bzw. Werte für einen oder mehrere Browser unterschiedlich notieren.

Alle CSS-Eigenschaften, die mit einem solchen herstellerspezifischen Präfix beginnen, sind NICHT VALIDE d.h der W3C-Validator kennt sie nicht und markiert sie als Fehler. Diesen Umstand können sie ignorieren.
